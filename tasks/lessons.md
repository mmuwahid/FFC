# FFC Lessons Learned

FFC inherits **all lessons** from `Padel Battle APP/tasks/lessons.md` — read that before writing any FFC code. Every Critical Rule there applies here too.

## Critical Rules (Always Enforce — inherited from PadelHub)
1. Mockup-first workflow. HTML mockup → user review → finalize → implement.
2. No design changes without explicit approval.
3. Placeholder data only in mockups. No invented names / leagues / stats.
4. Version control planning docs. V1, V2, V3. Never overwrite.
5. Never regenerate user documents. Edit originals.
6. Match existing app styling in mockups. Copy CSS from most recent approved mockup.
7. Safe-area insets for all fixed-position elements on mobile.
8. Verify before pushing fixes: run dev server, check console, visually verify.
9. When adding inline state-driven UI, declare `useState` first. Grep to confirm.
10. CRITICAL: verify DB columns exist before any Supabase `.select()` / `.update()`.
11. Never use `useMemo` for React Context with derived state. Plain object.
12. Never place hooks after conditional returns.
13. Global `<style>` must live in main render, not early-return subtree.
14. SECURITY DEFINER RPCs for all Edge Function DB access (never service-role + direct queries).
15. RPC parameters receiving JSON should be typed `TEXT`, cast to `JSONB` internally.
16. After `CREATE TABLE` via SQL editor: `GRANT SELECT, INSERT, UPDATE, DELETE ON public.X TO authenticated, anon;`
17. Always include `window.location.search` in `redirectTo` URLs (OAuth + reset).
18. Always scope `.delete()` / `.update()` with `.eq("league_id", ...)` — defense-in-depth beyond RLS.
19. Bump service-worker `CACHE_NAME` on every major deploy. Format: `ffc-vN`.
20. Full Windows path in Node scripts: `C:/Users/User/AppData/Local/Temp/FFC/...` — never `/tmp/`.
21. CRLF-aware string replacements on Windows git clones.
22. After triggering React state changes in `preview_eval`, wrap DOM reads in a 200–300ms timeout.
23. Git identity for FFC commits: `git -c user.name="Mohammed Muwahid" -c user.email="m.muwahid@gmail.com"`.

## FFC-specific lessons

### Mistakes

| Date | Mistake | Root Cause | Prevention Rule |
|------|---------|------------|-----------------|
| 20/APR/2026 (S008) | Five approved mockups (Poll v3 · Leaderboard v2 · Profile v3 · Captain helper v1 · Welcome — signed off across S005–S007) lack `viewport-fit=cover` meta + `env(safe-area-inset-*)` CSS despite Critical Rule #7 committing us to safe-area handling. Would render behind the iPhone notch / Dynamic Island / home-indicator in production PWA install. | Critical Rule #7 was treated as an implementation-phase rule; mockup review process never verified it. Desktop-browser phone-frames don't simulate the notch, so the gap was invisible during review. | **Mockup review checklist MUST verify: (1) `<meta name="viewport" content="…viewport-fit=cover">`; (2) root `--safe-*` custom properties (hardcoded simulation values — see next row); (3) `padding-top: var(--safe-top)` on `.topbar`; (4) `padding-bottom: var(--safe-bottom)` on `.bottom-nav` and fixed CTA stacks; (5) sheets/modals pad internal scroll with `calc(var(--safe-bottom) + 16px)`. Also — mockup phone-frame CSS MUST include a simulated iPhone-14-Pro Dynamic-Island cutout so the obstruction is visually obvious at review time. Reference: `docs/platform/iphone-safe-area.md`.** |
| 20/APR/2026 (S009) | First safe-area retrofit pass used `env(safe-area-inset-top, 59px)` expecting `59px` to apply in desktop preview. It didn't — Dynamic-Island cutout visibly overlapped the "Leaderboard" and "Pick captains" title text because `padding-top` never pushed content down. Two review cycles wasted. | The second argument of `env()` is a fallback used ONLY when the environment variable is **unrecognised** (ancient browser, misspelled name). Modern desktop browsers DO recognise `safe-area-inset-*`; they just resolve it to `0px` on rectangular viewports. Resolved-to-0 is a defined value, so the fallback never fires. I mis-read the CSS spec as "fallback when 0" and didn't verify in the preview panel. | **In MOCKUP files (desktop review artifacts only), hardcode simulation values directly: `--safe-top: 59px; --safe-bottom: 34px; --safe-left: 0px; --safe-right: 0px;`. Production app CSS is a separate stylesheet that reads `env(safe-area-inset-*)`. Do NOT try to make one CSS expression serve both contexts — the mockup and the deployed PWA have different requirements. When using `env()` anywhere, read the spec on what triggers the fallback BEFORE relying on it. Verify visually in preview after every safe-area change — do not mark a retrofit "done" from grep results alone.** |
| 20/APR/2026 (S009) | Safe-area v1 on statusbar used `padding-top: var(--safe-top)`. Pushed EVERYTHING (including the time/5G/battery) below the Dynamic Island — the statusbar glyphs disappeared into the content area and the top of the phone looked empty around the island. User caught it immediately: "true iOS apps have time on the left and signal/battery on the right OF the island, not below it." | iOS actually renders the system statusbar so time/battery FLANK the Dynamic Island at the same vertical level. `padding-top: var(--safe-top)` is correct for content cards that sit below the safe area, but WRONG for the statusbar itself — the statusbar needs to OCCUPY the safe area, with its content positioned to the left and right of the island. | **Statusbar v2 pattern (mockup + PWA): `.statusbar { height: var(--safe-top); display: flex; align-items: flex-end; justify-content: space-between; padding: 0 calc(28px + var(--safe-right)) 8px calc(28px + var(--safe-left)); }` — `align-items: flex-end` + `padding-bottom: 8px` positions time/battery 8px below the island's bottom edge with ~8px breathing room. Content cards BELOW the statusbar still use `padding-top: var(--safe-top)` pattern — they're different contracts.** |
| 21/APR/2026 (S010) | Subagent-applied flex-column + min-height fix on player profile KPI/achievement tiles rendered correctly in isolation — `.kpi` was `display: flex; min-height: 52px` and labels rendered at 12px height. But user saw text cut-off anyway in BOTH player profile and formation mockups. `preview_inspect` on live DOM showed `.statusbar { height: 25px }` instead of the declared `var(--safe-top) = 59px`. | `.phone-inner` is `display: flex; flex-direction: column`. `.statusbar` is the first flex child. When phone content is long enough to overflow the fixed 844px phone height, flex's default `flex-shrink: 1` shrinks the statusbar from 59px → 17–25px. That pulls every element up the page, and the first content block (team-strip / topbar) then sits BEHIND the Dynamic Island cutout. The "butchered UI" user perceived was actually the whole layout shifted up by ~40px from statusbar compression — NOT a cutoff in the tiles themselves. Only formation + profile hit it because they're the only two screens with enough content to overflow. Short-content mockups (Poll, Leaderboard, Settings, Admin Matches, etc.) rendered correctly because flex never had to shrink. | **Statusbar v2.1 amendment: add `flex-shrink: 0` to the `.statusbar` rule (or `flex: 0 0 var(--safe-top)`). Forces the statusbar to hold its full `--safe-top` height regardless of content length. Apply to ALL phone-frame mockups — not just the two that surfaced the bug. Diagnostic: when safe-area cutoff is reported despite correct CSS, first live-inspect `.statusbar` computed `height` — if it's less than `--safe-top`, flex-shrink is the culprit.** |
| 21/APR/2026 (S012) | S011's statusbar `flex-shrink: 0` fix verified fine on 59px statusbar. User re-verified Profile + Formation anyway and reported cards STILL looked butchered. Live DOM inspection revealed: `.card` rendering at 49px vs natural 185px in Profile; `.pitch-wrap` at 112px vs natural 505px + every other direct child of `.phone-inner` compressed 3–5× in Formation. The S011 fix addressed only ONE flex child (`.statusbar`) when the bug class applies to every child of the scroll container. | Same root cause as S010/S011 but generalised. `.phone-inner` is a scroll container with `overflow-y: auto`, and its `display: flex; flex-direction: column` treats EVERY direct child — not just the first one — as a shrinkable flex item. Whenever a child has `overflow: hidden` or `overflow: auto`, the flex spec forces its `min-height: auto → 0`, enabling compression to literally 0. Profile's `.card` has `overflow: hidden` for the border-radius clipping; Formation's `.pitch-wrap`, `.gk-card`, and `.roster` all have `overflow: hidden` for the same reason. That's why BOTH screens hit it despite S011's fix working on the statusbar. Only surfaces on screens whose natural content height exceeds the 844px phone — Profile (~1100px) and Formation (~1490px) were affected, other mockups weren't because their content fit. | **Statusbar v2.2 amendment — extend `flex-shrink: 0` from `.statusbar` to ALL direct children of `.phone-inner`. One defensive CSS selector covers it: `.phone-inner > * { flex-shrink: 0; }`. Apply as part of the mockup-frame boilerplate alongside the statusbar safe-area rules. Rationale: `.phone-inner` is a scroll container, not a stretch container — there's no case where a child of a scrolling phone frame should shrink below its natural size. Diagnostic when a content block appears clipped: live-inspect the child's `getBoundingClientRect().height` vs its `scrollHeight`; if h << scrollH, it's flex-shrink compression.** |
| 21/APR/2026 (S012) | Player Profile's bottom tab bar (`.tabbar`) used `position: absolute; bottom: 0; left: 0; right: 0` with `padding-bottom: 110px` on `.phone-inner` to reserve space for it. While the S012 statusbar-v2.2 flex fix was un-compressing the cards, the tabbar suddenly appeared in the MIDDLE of the scroll viewport followed by "Recent matches" below it — despite being the last child in the DOM. User caught it via screenshot. | `position: absolute; bottom: 0` on a child of a scroll container anchors to the bottom of the scroll **content**, not the viewport. Before S012's fix, content was flex-compressed to fit inside 824px so scroll-content-bottom coincidentally equaled viewport-bottom — masking the anchoring bug. After S012 uncompressed the cards, content became ~1318px. The absolute tabbar stayed anchored at y=1318 (the REAL content bottom), which is ~500px below the viewport's visible bottom when scrolled to the top of the screen. The user-observed behaviour: tabbar appears mid-viewport, Recent matches appears below it, both scroll away as you scroll down. | **Sticky-tabbar pattern for bottom navs in scroll containers: `.tabbar { position: sticky; bottom: 0; margin-top: auto; flex-shrink: 0; z-index: 10; /* rest unchanged */ }`. Remove any leftover `padding-bottom: Npx` on `.phone-inner` that was reserving space for the old absolute-positioned tabbar — sticky is in-flow and reserves its own space. Also: after ANY fix that changes a scroll container's content height, re-inspect every `position: absolute/fixed` child that may have been relying on the broken layout. Absolute positioning inside a scroll container is usually the wrong choice — sticky or a sibling-outside-the-scroller are almost always correct.** |

### Patterns That Work

- [20/APR/2026] (S008) **Frame decisions before drafting when multiple downstream artifacts depend on them.** S008 deliberately did not start the Settings spec until the user had answered team-colour + §3.15 sub-Qs + Settings Qs. Option A for §3.7 materially changes state 8 of the poll-screen spec — drafting under the deferred-assumption would have wasted ~30 min of spec+mockup re-work. **Why worth remembering:** framing feels slow in the moment but compounds into faster drafting. One 15-min framing session saves 30+ min of re-draft.
- [20/APR/2026] (S008) **When a user flags a cross-cutting platform concern mid-session, research it before closing — even if the session was otherwise done.** User raised the iPhone notch concern after saying "log this session." Doing the 15-min web research + mockup audit before closing converted a vague worry into an actionable S009 retrofit with a documented CSS pattern and enumerated file list. Closing without it would have left a known defect buried under "nice to have." **Why worth remembering:** research-before-close trades 15 min now for 60+ min of cold-start research on next session.

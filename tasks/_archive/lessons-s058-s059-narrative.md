## S058 patterns (issue close sweep)

- **Squash-merge of long-lived branches can silently delete unrelated changes (S058).** PR #28 (Atomo's roster-setup-v2 in S057) deleted 357 lines of S056 Payments page CSS that was added later than the PR's branch point. Squash merge resolved the merge by writing the post-squash file = pre-squash branch state in that range — overwriting the S056 additions. **Mitigation:** before merging any squash that touches a high-traffic file, run `git diff main..pr --stat` and grep for unexpected deletions. Add a "lines deleted" sanity check to PR review even when the PR only adds new code in its diff. **Restore pattern:** `git show <known-good-hash>:path | sed -n 'N,Mp' >> target` with banner comment — don't rewrite, don't refactor; rewriting introduces drift.

- **`ALTER TYPE … ADD VALUE` must be in its own BEGIN/COMMIT block (S058, reaffirms S043+S045).** Postgres requires the new enum value to be committed before it can appear in a function body or trigger. Pattern: `BEGIN; ALTER TYPE … ADD VALUE …; COMMIT; BEGIN; CREATE OR REPLACE FUNCTION … (uses new value); COMMIT;`. Missed once in the first draft of S058 mig 0057 (single transaction with both ADD VALUE and the new function); caught at write-time before push. **Generalises:** any migration that ALTERs a type AND uses the new value/column must split into ≥ 2 transactions.

- **CASCADE vs NO ACTION FK cleanup before DELETE (S058).** When writing an admin-delete RPC, query `pg_constraint` for child tables first: `SELECT conname, conrelid::regclass, confdeltype FROM pg_constraint WHERE contype = 'f' AND confrelid = '<target>'::regclass`. `confdeltype = 'c'` cascades automatically; `confdeltype = 'a'` (NO ACTION) blocks the DELETE with FK violation. **Pattern:** explicit `DELETE FROM <child> WHERE <fk> = <target>` for each NO-ACTION child, BEFORE the `DELETE FROM <target>`. Used in `admin_delete_match` for `match_payment_records` + `payment_windows`.

- **Re-snapshot ranks after any leaderboard-shifting admin action (S058).** When a leaderboard depends on a snapshot table for diff-based notifications, the snapshot must be refreshed after every admin action that mutates the underlying data — even if the LIVE leaderboard view recomputes correctly on read. Otherwise the next legitimate trigger (e.g., next match approval) will compare current state against a stale baseline and emit incorrect `ranking_changed` notifications. **Pattern:** `PERFORM snapshot_and_diff_ranks(season_id)` at the end of every admin RPC that touches `match_players` or `matches.result`.

- **Live-DB diagnosis BEFORE assuming a frontend bug (S058).** Issue #25b "matches tab is empty" — instinct was to debug Matches.tsx query. Live DB query showed 2 valid approved non-friendly matches that WOULD satisfy the existing query. Real cause was likely a friendly match the user saw in Profile (no filter) but which Matches tab filtered out. **Pattern:** for any "no data" report, query the live DB with the EXACT predicates the frontend uses BEFORE inspecting the frontend code. Saves over-engineering.

- **Open-an-EF-to-all-auth-users via RPC body change, not EF code change (S058).** S054's `render-match-card` EF was admin-gated via the RPC it calls (`get_match_card_payload` raised `Admin role required`). To open it to all authenticated players (for issue #25's PNG hero in MatchDetailSheet), changed the RPC body via migration 0056 — `IF NOT is_admin()` → `IF auth.uid() IS NULL`. The EF itself didn't change; no redeploy needed. The error-mapping line in the EF (`/Admin role required/i.test(rpcErr.message) ? 403`) just stops triggering. **Generalises:** when an EF's auth gate lives in the RPC it calls, opening the gate is a migration change, not an EF change.

- **Mockup-first delivers even when implementation seems obvious (S058).** Issue #21 had 4 "obvious" sub-asks (action grid + delete + edit roster + search). Building `admin-matches-v2.html` first forced the layout decisions before code, and the user caught a real gap — "what if scorer scored more than 1 goal" — via mockup review. Caught once, fixed in 30 lines of mockup HTML; would have been 200+ lines of React + sheet wiring + per-row goals integration if I'd skipped the mockup. **Generalises:** mockup-first per CLAUDE.md rule #1 has zero exceptions, even for "internal" admin screens.

- **The "scorer picker" approval was a UX shortcut interpretation, not a literal issue ask (S058).** Issue #21 text emphasized search on the *roster-add* picker, not scorers. The mockup tile B (scorer picker with tap-to-increment) was an interpretation extension. The existing per-row goals input on `match_players` rows already handles multi-goal cases adequately. **Pattern:** when a mockup tile is approved but the underlying issue text doesn't require it, flag as follow-up rather than blocking the parent work. Documented as S059 follow-up so it doesn't get lost OR over-prioritized.

- **Scope-creep detection during mockup feedback (S058).** User said "all approved but for the scorer picker what if he scored more than 1 goal" — single-thread modification. I added tap-to-increment ONLY to that tile, didn't redesign tile A (action grid) which user had already approved. **Pattern:** when user approves N-1 tiles and asks for change on tile X, scope changes to tile X; do not opportunistically rework other tiles unless explicitly invited.

## S059 patterns (live-debug cascade)

- **For per-row order in PG, never rely on `created_at` to disambiguate within a transaction (S059).** All FOREACH INSERTs in a single SECURITY DEFINER RPC share `now()` (transaction-start time). `ORDER BY created_at, id` falls back to uuid `id` for tiebreak — random. **Pattern:** if you need stable insertion order for display, add an explicit `slot_index integer` column written by the RPC at insert time. Backfill existing rows via `ROW_NUMBER() OVER (PARTITION BY <group> ORDER BY created_at, id)` (best-effort; arbitrary for transaction-mate rows but deterministic going forward). Generalises beyond `match_players` — anywhere bulk INSERT order matters for display.

- **For destructive admin RPCs, ask "what does the user expect this to delete" not "what does the FK graph let me delete" (S059).** Migration 0061's `admin_delete_matchday` refused on any `matches` row — wrong mental model — fixed in 0062 to cascade through UNAPPROVED draft matches and only refuse on APPROVED matches. **Default rule:** cascade-through-children for non-load-bearing children (drafts, votes, tokens). Only refuse on children that represent committed history (approved matches, payment records).

- **`migration repair --status applied <version>` to record a remote migration that was applied without local tracking (S059).** Stray `0060_seasons_games_seeded.sql` had been applied from `bb13127` but local history didn't know. `npx supabase db push` errored on duplicate version key (23505). `npx supabase migration repair --status applied 0060 --linked` recorded it without re-running. **Less destructive than dropping/re-applying;** preserves actual remote state.

- **`CREATE OR REPLACE FUNCTION` to re-issue an RPC body in a follow-up migration (S059).** When args unchanged, no DROP needed → keeps upgrade path linear without breaking GRANT EXECUTE state. Used 4× in S059 (0062 updating admin_delete_matchday body; 0063 updating create_match_draft + admin_update_match_draft + admin_edit_match_roster bodies to write slot_index). Reaffirms S028's "CREATE OR REPLACE cannot change arg defaults or add parameters" — corollary: if args ARE unchanged, OR REPLACE is the right tool.

- **`WITH ORDINALITY` + `ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ord)` for slot_index in jsonb-array RPCs (S059).** `admin_edit_match_roster` accepts a jsonb array; `WITH ORDINALITY` exposes input position as `ord`, then PARTITION BY team gives per-team 1-based slot_index. Single-statement insert, no procedural counter. **Clean SQL idiom matching how FOREACH counters work in plpgsql peers** — same `slot_index` semantics across 3 RPCs (`create_match_draft`, `admin_update_match_draft`, `admin_edit_match_roster`).

- **Always grep for sibling usages when fixing a Supabase query bug (S059).** Poll.tsx had a missing `.eq('match_id', …)` filter on its `match_players` query — bug returned every row across every match each player had been in, then `mpMap.set(profile_id, …)` overwrote randomly. Audit via `grep -rn "from('match_players')"` confirmed FormationPlanner / MatchDetailSheet / AdminMatches / AdminRosterSetup / CaptainHelper / Profile already scoped correctly. **Confirms the bug is local rather than systemic** and prevents speculative refactors.

- **Verify hypothesis with DOM eval before coding the fix (S059).** Phase A debug: tapped Review/approve in user's screenshot → opened EditResultSheet not MatchEntryReview → that 1 fact reframed the entire diagnosis (it's a `matches` row, not a `pending_match_entries` row). Schema technically allowed both interpretations; **UI-state evidence trumps schema reading** when the schema is ambiguous about which path triggered.

- **User asks "where is the X feature" → check the gating predicate, not the code path (S059).** Both ref-link button and Delete-match button were guarded by predicates that didn't match the user's actual state (`{locked && ...}` and `{approved && md.match}` respectively). Code existed; gates were wrong. **Pattern:** when a user reports "I can't see X", first check WHEN X is supposed to render (the predicate), not the underlying logic.

§3.18 Admin Matches · v1 design notes (read before approval)

  • Phase indicator drives the card. The phase chip is the single source of truth for what the admin should do next — Poll open (no CTA) → Lock roster pending (warn amber) → Roster locked — pick captains (accent) → Captains set — enter teams → Result pending → Final (success green). Each phase exposes exactly one primary CTA.
  • Captain picking lives in §3.1-v2, not here. The Phase 3 CTA deep-links to §3.1-v2 with the matchday_id query param and returns here on back. set_matchday_captains(matchday_id, white, black) is the new RPC (S009 addition to §2.7).
  • Guest-stats correction is long-press, not tap. Tap on a guest row is a regular navigation action (opens guest profile view, Phase 2); long-press opens the correction sub-sheet. Discoverable via an "Edit guest" overflow affordance on each guest line in Results-pending expanded view.
  • Edit-after-approval writes audit. edit_match_result(match_id, patch) writes the full before/after diff to admin_audit_log and cascades to v_season_standings. No silent overwrites — the standings-refresh toast confirms propagation.
  • Late-cancel penalty edits go through edit_match_result. No dedicated penalty-edit UI in Phase 1; admin zeroes late_cancel_points_delta in the match patch if a penalty was applied in error. Dedicated UI deferred.
  • Safe area. Dynamic Island cutout visible at the top of both phone frames; tabbar padding uses var(--safe-bottom) per §3.0.

State 1 · First-visit push prompt

In-page permission prompt (before OS prompt)

Renders at the top of the Settings screen on first visit while Notification.permission === 'default'. Controls the first surface so the player reads copy before the browser's OS prompt fires — avoids the "user clicks No reflexively" failure mode. Dismissible for the session; re-renders on next visit until explicit grant or deny.

🔔 Stay in the loop
Enable push to know when polls open and teams are revealed. Four notifications a week, max.
State 2 · Push denied fallback

Persistent re-enable instructions

Renders when Notification.permission === 'denied'. Persistent — cannot be dismissed (the information is load-bearing every visit the player returns hoping to enable push). Row 2's master toggle is forced OFF and non-interactive while this tile is present — JS can't re-prompt once denied, so hiding the broken affordance prevents confusion.

⚠️ Push is blocked by your browser
To enable FFC push notifications:
  1. Tap the 🔒 lock icon in your browser's address bar.
  2. Find Notifications and switch to Allow.
  3. Refresh this page.

§3.16 Settings · S009 round-2 amendments · mockup v2 for review

  • Six rows, locked order: Theme · Push notifications · Leaderboard sort · Positions · Display name · Account. Phase-2 extras (About/version, T&P links, Data export, functional Delete) are NOT rendered.
  • Theme default = Dark (round-2 change). User can switch to Light or System at any time; selection auto-saves and persists per-device.
  • Push-prefs defaults (round-2): master ON · poll_open ON · poll_reminder ON · fires 2 min before poll close · roster_locked ON · plus_one_unlocked ON · match_result_posted ON · dropout_after_lock ON (NEW). position_changed removed entirely — admin-confirmation event no longer user-configurable.
  • Dropout-after-lock notification: fires to all confirmed-roster players (+ admins) when a player cancels after roster lock and a substitute is promoted from the waitlist. Event-wiring lives in Section 5.
  • Poll-reminder timing = 2 min before close: intentional "last call to vote" behaviour. Very tight window — if silent-device players complain post-launch, Section 5 can widen or make user-configurable in Phase 2.
  • Master OFF ≠ reset. Greying out child checkboxes preserves stored values. Re-enabling master restores prior selection.
  • Display-name validation: ^[A-Za-z0-9 .'-]{2,30}$. Save disabled while value equals stored, while invalid, and during in-flight. Uniqueness check (case-insensitive) on save — conflict shakes the field with inline error "That name's taken."
  • Delete account · coming soon. Row renders muted, non-functional. Tap shows a "Phase 2" toast. Deferred to Phase 2 with 30-day soft-delete + admin confirmation chain.
  • Permission coupling: toggling master ON while Notification.permission === 'default' fires Notification.requestPermission(). Deny → surface State tile 2 and revert master to OFF. Grant → register push subscription and save.
  • Leaderboard sort UX: selection applies on the next Leaderboard entry, not mid-session — a visible "applied on next visit" hint is present so the behaviour isn't a surprise.
  • Sign out: clears session + deregisters this device's push subscription endpoint + returns to auth screen. Other devices keep their subscriptions.

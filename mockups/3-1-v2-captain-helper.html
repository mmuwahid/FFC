§3.1-v2 decisions (cross-ref design spec §3.1-v2)

  • Mode default. season_matchdays_approved ≥ 5 → formula default; < 5 → randomizer default. Visible toggle lets the admin override in either direction.
  • Three criteria (S004 simplification). Matches ≥ 5 · attendance ≥ 60% · cooldown ≥ 4 matchdays since last captained. Dropped V2.0's "knows the group" and "pair balance" criteria — first was expensive to compute, second applies at the pair level, not the candidate level.
  • Pair balance. Rank gap ≤ 5 = green ✓ ok. Rank gap > 5 = amber ⚠ warning with "Proceed anyway?" sub-modal — not a hard block.
  • White = weaker (S004 rule). Auto-applied in the pair-confirmation sheet. The lower-ranked player in the pair (higher position number) captains White and picks first. Not overridable in Phase 1.
  • Guest handling (S007). Guests appear in a read-only subsection with all S007 stat fields (pills · rating · stamina/accuracy · description). Cannot be selected as captain. Their presence on the roster is surfaced so the admin can factor them into pair balance.
  • Write path. Confirming a pair fires set_matchday_captains(matchday_id, white_profile_id, black_profile_id) — new RPC to be added to §2.7 Part 5B. Writes audit columns matches.captain_assigned_at + matches.captain_assigned_by.
  • Concurrency. Two admins can open the screen concurrently. Last write wins; later admin sees a toast ("Captains were picked by Fatima 30s ago"). Captain pair is editable post-confirmation via "Edit" on the matchday screen.
  • Phase 2 deferred. Auto-pick + 60-min override window on roster_locked. Async captain draft entrypoint. Per-player goal-difference as 4th criterion when match_players.goals_against is tracked.

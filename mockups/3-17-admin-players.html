§3.17 Admin Players · v1 design notes (read before approval)

  • Role tier safety. The overflow-menu action Promote to admin is visible only when the viewing admin is super_admin. Promotion to super_admin is intentionally absent from the UI in Phase 1 — documented as a manual DB-level action — so a compromised admin account can't self-elevate.
  • Reject requires a reason. No silent rejection. The reason is written to profiles.reject_reason (new column, §2.2 amendment) and audit-logged. Player is not notified of the rejection; they see an error banner on next sign-in attempt.
  • Ban is matchday-scoped, not date-scoped. until_matchday_id binds to a specific future matchdays row, so a rescheduled matchday automatically moves the return date. Dates render DD/MMM/YYYY uppercase.
  • Audit log is non-optional. Every write (approve_signup, reject_signup, update_player_profile, ban_player, unban_player, role changes) logs to admin_audit_log — masterplan V2.7 will add this table to §2.3.
  • Segment counts are live. Pending (3) reflects the true row count and drives the admin's attention directly to the queue that needs clearing.
  • Safe area. Status-bar padding uses var(--safe-top); tabbar uses var(--safe-bottom). Dynamic Island cutout is rendered at top: 21px via .phone::after so obstruction is visible during review.
